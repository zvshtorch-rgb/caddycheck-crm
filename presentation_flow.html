The 4 Layers at a Glance

Streamlit
Web UI + business logic. Runs in the browser, reads & writes data.
🗄️
Supabase
PostgreSQL database + file storage buckets. All persistent data lives here.
🐙
GitHub
Source code repository. Triggers automatic redeploy on every push.
☁️
Streamlit Cloud
Hosts the app publicly. Injects secrets, installs dependencies, serves users.

⚠️ Secrets (Supabase URL & keys) are set in Streamlit Cloud dashboard — never committed to Git.

How Secrets Are Handled

# Local development: .streamlit/secrets.toml (NOT in Git)
[supabase]
url             = "https://xxxx.supabase.co"
service_role_key = "eyJ..."
# supabase_service.py reads them at runtime:
cfg = st.secrets.get("supabase", {})
url = cfg.get("url", "")
key = cfg.get("service_role_key", cfg.get("anon_key", ""))
On Streamlit Cloud, the same values are entered in App Settings → Secrets (UI, not a file).

Full Lifecycle – Step by Step

1
Developer writes code locally

Edits streamlit_app.py, services/, ui/ etc. on their machine.

2
git push → GitHub

Code is pushed to the GitHub repository (no data, no secrets).

3
Streamlit Cloud detects the push

Automatically pulls the new code, installs requirements_streamlit.txt, restarts the app.

4
User opens the app URL

Streamlit renders the UI; the app connects to Supabase using injected secrets.

5
Data read/write goes to Supabase

All projects, invoices, PDFs, and logs are stored and retrieved from Supabase — not from Git.

Key Principle

🐙
Git = Code
Source code, UI, business logic, migrations.

NOT data, NOT secrets.
🗄️
Supabase = Data
All persistent data: customers, invoices, PDFs, logs.

NOT code, NOT deployments.
